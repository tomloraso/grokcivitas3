## Objective

Introduce a pre-generated, factual school overview (120-180 words) for every school, using a port-based LLM integration that follows the existing hexagonal architecture. This is the free-tier AI feature described in `.planning/ai-features.md`.

## Scope

### In scope

- Domain port protocol for summary generation.
- Application use case for generating and retrieving school overviews.
- Infrastructure LLM adapter (swappable provider).
- Prompt template for factual school overview.
- Data-version-hash cache to detect stale summaries.
- Database storage of generated text with provenance metadata.
- Operational integration for batch pre-generation after successful Gold refresh.
- API exposure and frontend display.

### Out of scope

- Premium "Grok's Take" analysis (owned by `AI-3`).
- Live/on-demand generation at request time.
- Comparison narratives and NL search (owned by `AI-4`).

## Design Decisions

1. **Pre-generated, not live.** Summaries are batch-generated by a dedicated AI generation job after data pipeline success and stored in the database. User requests return cached text instantly.
2. **Port-based LLM integration.** The application layer defines a `SummaryGenerator` protocol; infrastructure provides the concrete LLM adapter. Provider is swappable without application changes.
3. **Data-version hash.** Each summary is keyed to a hash of the input data (school fields + demographics + performance + Ofsted). When upstream data changes, the summary is marked stale and queued for regeneration.
4. **Facts only.** The overview prompt enforces neutral, factual language. No opinions, rankings, subjective advice, or recommendations.
5. **Model choice.** Primary model: configurable via settings (default: Grok 4.1 Fast Reasoning per `.planning/ai-features.md`). Model selection is infrastructure config, not application logic.
6. **Versioned prompts.** Prompt templates are version-controlled Python modules, not database-stored strings.
7. **Cost model.** ~25,000 schools x 2 refreshes/year at Grok 4.1 pricing is expected to stay under GBP 50/year.
8. **Provider selection via config.** `CIVITAS_AI_PROVIDER` selects an infrastructure adapter (`grok`, `openai`, or `openai_compatible`). Bootstrap resolves the adapter through a factory without changing application code.

## Data Assembly

The overview prompt receives a structured data context assembled from Gold tables:

```
SchoolOverviewContext:
  # From schools (GIAS enriched - AI-1)
  urn, name, phase, school_type, status, postcode
  website, telephone, head_name, head_job_title
  statutory_low_age, statutory_high_age, gender
  religious_character, admissions_policy, sixth_form
  trust_name, la_name, urban_rural
  pupil_count, capacity, number_of_boys, number_of_girls

  # From school_demographics_yearly (latest year)
  fsm_pct, eal_pct, sen_pct, ehcp_pct

  # From school_performance_yearly (latest year)
  progress_8, attainment_8, ks2_reading_met, ks2_maths_met

  # From school_ofsted_latest
  overall_effectiveness, inspection_date

  # From area_deprivation
  imd_decile
```

## Data Version Hash

```python
hash = sha256(
    urn + "|" +
    json.dumps(sorted_data_context, sort_keys=True)
).hexdigest()
```

- Stored alongside the generated text.
- On each pipeline run, recompute hash for each school.
- If hash differs from stored hash, mark summary as stale.
- Stale summaries are regenerated in the next batch run.

## Operational Integration And Run Semantics

AI summary generation is explicitly a **post-promote** operation:

1. Bronze -> Silver -> Gold runs complete first using the existing pipeline runner.
2. AI generation executes only after required source runs are `succeeded` or `skipped_no_change`.
3. AI generation reads Gold tables, computes current hashes, and upserts stale summaries.
4. AI generation failures do not silently pass; the command exits non-zero with failure counts.
5. Generation is resumable (`--resume-run-id`) and idempotent (current hashes are skipped).

This keeps the core data pipeline contract unchanged while providing deterministic AI refresh behavior.

## Prompt Template Structure

```
prompt_templates/
  school_overview.py
    SYSTEM_PROMPT: str  - role, constraints, output format
    USER_TEMPLATE: str  - Jinja2-style template with data slots
    VERSION: str        - semantic version for prompt tracking
```

Prompt constraints:
- 120-180 words.
- Must cover: school type, age range, size, location character, key strengths from data.
- Must not: rank against other schools, give advice, use superlatives, speculate.
- Must not: reference data it was not given (no hallucinated facts).

## Database Schema

New table: `school_ai_summaries`

```sql
CREATE TABLE school_ai_summaries (
    urn text NOT NULL,
    summary_type text NOT NULL,          -- 'overview' | 'groks_take'
    text text NOT NULL,
    data_version_hash text NOT NULL,
    prompt_version text NOT NULL,
    model_id text NOT NULL,
    generated_at timestamptz NOT NULL,
    generation_duration_ms integer NULL,
    PRIMARY KEY (urn, summary_type)
);
CREATE INDEX ix_school_ai_summaries_stale
    ON school_ai_summaries (data_version_hash);
```

Design notes:
- Composite PK `(urn, summary_type)` supports overview and premium tiers in one table.
- `prompt_version` and `model_id` provide full provenance for audit.
- Previous versions are not retained in this table (simplicity first; add history table in `AI-4` if needed).

## Hexagonal Architecture Mapping

### Domain layer

`apps/backend/src/civitas/domain/school_summaries/`

1. `models.py` (new)
   - `SchoolSummary` frozen dataclass: `urn`, `summary_type`, `text`, `data_version_hash`, `prompt_version`, `model_id`, `generated_at`, `generation_duration_ms`.
   - `SchoolOverviewContext` frozen dataclass: assembled data context for prompt.
   - `SummaryType` literal: `"overview"`, `"groks_take"`.

### Application layer

`apps/backend/src/civitas/application/school_summaries/`

1. `ports/summary_generator.py` (new)
   - `SummaryGenerator` Protocol:
     ```python
     class SummaryGenerator(Protocol):
         def generate_overview(self, context: SchoolOverviewContext) -> GeneratedSummary: ...
     ```
2. `ports/summary_repository.py` (new)
   - `SummaryRepository` Protocol:
     ```python
     class SummaryRepository(Protocol):
         def get_summary(self, urn: str, summary_type: SummaryType) -> SchoolSummary | None: ...
         def upsert_summary(self, summary: SchoolSummary) -> None: ...
         def list_stale(self, current_hashes: dict[str, str]) -> list[str]: ...
     ```
3. `use_cases.py` (new)
   - `GetSchoolOverviewUseCase`: retrieve cached summary by URN.
   - `GenerateSchoolOverviewsUseCase`: batch generate/refresh overviews for stale schools.
4. `dto.py` (new)
   - `SchoolSummaryDto`: application-layer DTO for API mapping.

### Infrastructure layer

`apps/backend/src/civitas/infrastructure/ai/`

1. `provider_factory.py` (new)
   - `build_summary_generator(settings) -> SummaryGenerator`.
   - Maps `CIVITAS_AI_PROVIDER` to a concrete adapter.

2. `providers/grok_summary_generator.py` (new)
   - Implements `SummaryGenerator` for Grok endpoints.

3. `providers/openai_summary_generator.py` (new)
   - Implements `SummaryGenerator` for OpenAI endpoints.

4. `providers/openai_compatible_summary_generator.py` (new)
   - Implements `SummaryGenerator` for self-hosted OpenAI-compatible endpoints (for example local/open-source deployments).

Common adapter behavior:
- Use `httpx` transport at infrastructure layer only.
- Handle rate limiting, retries, and timeout.
- Model selection from settings.

5. `prompt_templates/school_overview.py` (new)
   - `SYSTEM_PROMPT`, `USER_TEMPLATE`, `VERSION` constants.
   - `render(context: SchoolOverviewContext) -> tuple[str, str]` function returning (system, user) messages.

6. `prompt_templates/__init__.py` (new)
   - Registry of prompt template modules by summary type.

`apps/backend/src/civitas/infrastructure/persistence/`

## Acceptance Criteria

1. `school_ai_summaries` table exists with composite PK `(urn, summary_type)`.
2. `SummaryGenerator` protocol is defined in application ports, not infrastructure.
3. LLM provider and model are swappable via settings without code changes.

5. Data-version hash correctly detects stale summaries after pipeline runs.

7. Profile API exposes overview text.
8. Frontend displays overview on school profile page.
9. `CIVITAS_AI_ENABLED=false` disables all AI generation (feature flag).

## Risks And Mitigations

- Risk: LLM API rate limits during batch generation of 25,000 schools.
  - Mitigation: configurable batch size, exponential backoff, resumable generation (skip already-current hashes).
- Risk: Generated text contains hallucinated facts not in the input data.
  - Mitigation: prompt explicitly constraints output to provided data only; integration tests validate no external claims.
- Risk: API key exposure in config.
  - Mitigation: key loaded from environment variable, never committed; settings validation fails fast on missing key when AI enabled.
- Risk: LLM providers differ in API behavior, rate limits, and output style.
  - Mitigation: provider-specific adapters behind a shared port, provider factory tests, and config-driven provider selection.
